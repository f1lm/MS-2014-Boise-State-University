A Simple Introduction to AOP

by Biju Kunjummen on June 21st, 2012 | Filed in: Enterprise Java Tags: AOP, AspectJ, Spring
Why use AOP, a simple way to answer this question is to show an implementation of a cross cutting concern without using AOP. Consider a simple service and it’s implementation:
 
 
 
 
 
 
 
 

view sourceprint?
1public interface InventoryService {
2    public Inventory create(Inventory inventory);
3    public List<inventory> list();
4    public Inventory findByVin(String vin);
5    public Inventory update(Inventory inventory);
6    public boolean delete(Long id);
7    public Inventory compositeUpdateService(String vin, String newMake);
8}

and its default implementation:

view sourceprint?
01public class DefaultInventoryService implements InventoryService{
02     
03    @Override
04    public Inventory create(Inventory inventory) {
05        logger.info("Create Inventory called");
06        inventory.setId(1L);
07        return inventory;
08    }
09 
10    @Override
11    public List<inventory> list(){
12        return new ArrayList<inventory>();
13    }
14 
15    @Override
16    public Inventory update(Inventory inventory) {
17        return inventory;
18    }
19 
20    @Override
21    public boolean delete(Long id) {
22        logger.info("Delete Inventory called");
23        return true;
24    }
25....

This is just one service. Assume that there are many more services in this project.

So now, if there were a requirement to record the time taken by each of the service methods, the option without AOP would be something along the following lines. Create a decorator for the service:

view sourceprint?
01public class InventoryServiceDecorator implements InventoryService{
02    private static Logger logger = LoggerFactory.getLogger(InventoryServiceDecorator.class);
03    private InventoryService decorated;
04 
05    @Override
06    public Inventory create(Inventory inventory) {
07        logger.info("before method: create");
08        long start = System.nanoTime();
09        Inventory inventoryCreated = decorated.create(inventory);
10        long end = System.nanoTime();
11        logger.info(String.format("%s took %d ns", "create", (end-start)) );
12        return inventoryCreated;
13    }

This decorator would essentially intercept the call on behalf of the decorated, record the time taken for the method call while delegating the call to the decorated object.

Imagine doing this for all the methods and all the services in the project. This is the scenario that AOP addresses, it provides a way for the cross cutting concerns(Recording time for the service method calls for eg.) to be modularized – to be packaged up separately without polluting the core of the classes.

To end the session, a different way to implement the decorator would be using the dynamic proxy feature of Java:

view sourceprint?
01public class AuditProxy implements java.lang.reflect.InvocationHandler {
02     
03    private static Logger logger = LoggerFactory.getLogger(AuditProxy.class);
04    private Object obj;
05 
06    public static Object newInstance(Object obj) {
07        return java.lang.reflect.Proxy.newProxyInstance(obj.getClass().getClassLoader(), obj
08                .getClass().getInterfaces(), new AuditProxy(obj));
09    }
10 
11    private AuditProxy(Object obj) {
12        this.obj = obj;
13    }
14 
15    public Object invoke(Object proxy, Method m, Object[] args) throws Throwable {
16        Object result;
17        try {
18            logger.info("before method " + m.getName());
19            long start = System.nanoTime();
20            result = m.invoke(obj, args);
21            long end = System.nanoTime();
22            logger.info(String.format("%s took %d ns", m.getName(), (end-start)) );
23        } catch (InvocationTargetException e) {
24            throw e.getTargetException();
25        } catch (Exception e) {
26            throw new RuntimeException("unexpected invocation exception: " + e.getMessage());
27        } finally {
28            logger.info("after method " + m.getName());
29        }
30        return result;
31    }
32}

So now, when creating an instance of InventoryService, I would create it through the AuditProxy dynamic proxy:

view sourceprint?
1InventoryService inventoryService = (InventoryService)AuditProxy.newInstance(new DefaultInventoryService());

the overridden invoke method of java.lang.reflect.InvocationHandler would intercept all calls to the InventoryService created in this manner, where the cross cutting concern of auditing the method call time is recorded. This way the cross cutting concern is modularized to one place(AuditProxy), but still needs to be explicitly known by the clients of InventoryService, when instantiating InventoryService.

Now, I will show how the cross-cutting concern can be implemented using Spring AOP – Spring offers multiple ways of implementing Aspects – XML configuration based, @AspectJ based. In this specific example, I will use XML configuration file based way of defining the aspect

Spring AOP works in the context of a Spring container, so the service implementation that was defined in the previous session needs to be a Spring bean, I am defining it using the @Service annotation:

view sourceprint?
1@Service
2public class DefaultInventoryService implements InventoryService{
3...
4}

Now, I want to record the time taken for each of the method calls of my DefaultInventoryService – I am first going to modularize this as an “advice”:

view sourceprint?
01package org.bk.inventory.aspect;
02 
03import org.aspectj.lang.ProceedingJoinPoint;
04import org.slf4j.Logger;
05import org.slf4j.LoggerFactory;
06 
07public class AuditAdvice {
08 
09    private static Logger logger = LoggerFactory.getLogger(AuditAdvice.class);
10 
11    public void beforeMethod() {
12        logger.info("before method");
13    }
14 
15    public void afterMethod() {
16        logger.info("after method");
17    }
18 
19    public Object aroundMethod(ProceedingJoinPoint joinpoint) {
20        try {
21            long start = System.nanoTime();
22            Object result = joinpoint.proceed();
23            long end = System.nanoTime();
24            logger.info(String.format("%s took %d ns", joinpoint.getSignature(), (end - start)));
25            return result;
26        } catch (Throwable e) {
27            throw new RuntimeException(e);
28        }
29    }
30         
31}

This advice is expected to capture the time taken by the methods in DefaultInventoryService. So now to wire this advice to the DefaultInventoryService spring bean:

view sourceprint?
01<bean id="auditAspect" class="org.bk.inventory.aspect.AuditAdvice" />
02 
03<aop:config>
04 <aop:aspect ref="auditAspect">
05  <aop:pointcut id="serviceMethods" expression="execution(* org.bk.inventory.service.*.*(..))" />
06 
07  <aop:before pointcut-ref="serviceMethods" method="beforeMethod" /> 
08  <aop:around pointcut-ref="serviceMethods" method="aroundMethod" />
09  <aop:after-returning pointcut-ref="serviceMethods" method="afterMethod" />
10 </aop:aspect>
11</aop:config>

This works by first defining the “pointcut” – the places(in this example, the service methods) to add the cross cutting concern(capturing the method execution time in this example) to. Here I have defined it using a pointcut expression –

view sourceprint?
1execution(* org.bk.inventory.service.*.*(..))

, which is essentially selecting all methods of all the types in the org.bk.inventory.service package. Once the pointcut is defined, it defines what needs to be done around the pointcut(the advice), using the expression:

view sourceprint?
1<aop:around pointcut-ref="serviceMethods" method="aroundMethod" />

This basically says, that around every method of any service type, execute the aroundMethod of AspectAdvice that was defined earlier. Now, if the service methods are executed, I would see the advice getting invoked during the method execution, the following is a sample output if DefaultInventoryService, createInventory method is called:

view sourceprint?
1org.bk.inventory.service.InventoryService - Create Inventory called
2org.bk.inventory.aspect.AuditAdvice - Inventory org.bk.inventory.service.InventoryService.create(Inventory) took 82492 ns

Spring’s AOP implementation works by generating a dynamic proxy at runtime for all the target beans, based on the defined pointcut.

Another way of defining an Aspect is using @AspectJ annotaions – which is natively understood by Spring:

view sourceprint?
01package org.bk.inventory.aspect;
02 
03import org.aspectj.lang.ProceedingJoinPoint;
04import org.aspectj.lang.annotation.After;
05import org.aspectj.lang.annotation.Around;
06import org.aspectj.lang.annotation.Aspect;
07import org.aspectj.lang.annotation.Before;
08import org.aspectj.lang.annotation.Pointcut;
09import org.slf4j.Logger;
10import org.slf4j.LoggerFactory;
11 
12@Aspect
13public class AuditAspect {
14 
15    private static Logger logger = LoggerFactory.getLogger(AuditAspect.class);
16 
17    @Pointcut("execution(* org.bk.inventory.service.*.*(..))")
18    public void serviceMethods(){
19        //
20    }
21 
22     
23    @Before("serviceMethods()")
24    public void beforeMethod() {
25        logger.info("before method");
26    }
27 
28    @Around("serviceMethods()")
29    public Object aroundMethod(ProceedingJoinPoint joinpoint) {
30        try {
31            long start = System.nanoTime();
32            Object result = joinpoint.proceed();
33            long end = System.nanoTime();
34            logger.info(String.format("%s took %d ns", joinpoint.getSignature(), (end - start)));
35            return result;
36        } catch (Throwable e) {
37            throw new RuntimeException(e);
38        }
39    }
40       
41    @After("serviceMethods()")
42    public void afterMethod() {
43        logger.info("after method");
44    }   
45}

The @Aspect annotation on the class identifies it as an aspect definition. It starts by defining the pointcuts:

view sourceprint?
1@Pointcut("execution(* org.bk.inventory.service.*.*(..))")
2    public void serviceMethods(){}

The above basically identifies all the methods of all types in org.bk.inventory.service package, this pointcut is identified by the name of the method on which the annotation is placed – in this case “serviceMethods”. Next, the advice is defined using the @Before(serviceMethods()), @After(serviceMethods()) and @Around(serviceMethods()) annotation and the specifics of what needs to happen is the body of the methods with those annotations. Spring AOP natively understands the @AspectJ annotations, if this Aspect is defined as a bean:

view sourceprint?
1<bean id="auditAspect" class="org.bk.inventory.aspect.AuditAspect" />

Spring would create a dynamic proxy to apply the advice on all the target beans identified as part of the pointcut notation.

Yet another way to define an aspect – this time using native aspectj notation.

view sourceprint?
01package org.bk.inventory.aspect;
02 
03import org.bk.inventory.types.Inventory;
04import org.slf4j.Logger;
05import org.slf4j.LoggerFactory;
06 
07public aspect AuditAspect {
08    private static Logger logger = LoggerFactory.getLogger(AuditAspect.class);
09 
10    pointcut serviceMethods() : execution(* org.bk.inventory.service.*.*(..));
11 
12    pointcut serviceMethodsWithInventoryAsParam(Inventory inventory) : execution(* org.bk.inventory.service.*.*(Inventory)) && args(inventory);
13 
14    before() : serviceMethods() {
15        logger.info("before method");
16    }
17 
18    Object around() : serviceMethods() {
19        long start = System.nanoTime();
20        Object result = proceed();
21        long end = System.nanoTime();
22        logger.info(String.format("%s took %d ns", thisJoinPointStaticPart.getSignature(),
23                (end - start)));
24        return result;
25    }
26 
27    Object around(Inventory inventory) : serviceMethodsWithInventoryAsParam(inventory) {
28        Object result = proceed(inventory);
29        logger.info(String.format("WITH PARAM: %s", inventory.toString()));
30        return result;
31    }
32    after() : serviceMethods() {
33        logger.info("after method");
34    }
35}

This maps to the previously defined @AspectJ notation

Since this is a DSL specifically for defining Aspects, it is not understood by the java compiler. AspectJ provides a tool(ajc) to compile these native aspectj files and to weave the aspects into the targeted pointcuts. Maven provides a plugin which seamlessly invokes ajc at the point of compilation:

This will be a wrap up of the AOP intro, with an example that will comprehensively exercise the concepts introduced in the previous sessions.

The use case is simple, I am going to define a custom annotation, PerfLog, I expect the calls to methods annotated with this annotation to be timed and logged. Let me start by defining the annotation:

view sourceprint?
01package org.bk.annotations;
02import java.lang.annotation.ElementType;
03import java.lang.annotation.Retention;
04import java.lang.annotation.RetentionPolicy;
05import java.lang.annotation.Target;
06 
07@Target({ElementType.TYPE, ElementType.METHOD})
08@Retention(RetentionPolicy.RUNTIME)
09public @interface PerfLog {
10     
11}

Now to annotate some service methods with this annotation:

view sourceprint?
01@Service
02public class DefaultInventoryService implements InventoryService{
03     
04    private static Logger logger = LoggerFactory.getLogger(InventoryService.class);
05 
06     
07    @Override
08    public Inventory create(Inventory inventory) {
09        logger.info("Create Inventory called");
10        inventory.setId(1L);
11        return inventory;
12    }
13 
14    @Override
15    public List<Inventory> list() {
16        return new ArrayList<Inventory>();
17    }
18 
19    @Override
20    @PerfLog
21    public Inventory update(Inventory inventory) {
22        return inventory;
23    }
24 
25    @Override
26    public boolean delete(Long id) {
27        logger.info("Delete Inventory called");
28        return true;
29    }
30 
31    @Override
32    @PerfLog
33    public Inventory findByVin(String vin) {
34        logger.info("find by vin called");
35        return new Inventory("testmake", "testmodel","testtrim","testvin" );
36    }
37 
38    @Override
39    @PerfLog
40    public Inventory compositeUpdateService(String vin, String newMake) {
41        logger.info("composite Update Service called");
42        Inventory inventory = findByVin(vin);
43        inventory.setMake(newMake);
44        update(inventory);
45        return inventory;
46    }
47}

Here three methods of DefaultInventoryService have been annotated with @PerfLog annotation – update, findByVin, compositeUpdateService which internally invokes the methods findByVin and update.

Now for the Aspect which will intercept all calls to methods annotated with @PerfLog and log the time taken for the method call:

view sourceprint?
01package org.bk.inventory.aspect;
02 
03import org.aspectj.lang.ProceedingJoinPoint;
04import org.aspectj.lang.annotation.Around;
05import org.aspectj.lang.annotation.Aspect;
06import org.aspectj.lang.annotation.Pointcut;
07import org.slf4j.Logger;
08import org.slf4j.LoggerFactory;
09 
10@Aspect
11public class AuditAspect {
12 
13    private static Logger logger = LoggerFactory.getLogger(AuditAspect.class);
14 
15    @Pointcut("execution(@org.bk.annotations.PerfLog * *.*(..))")
16    public void performanceTargets(){}
17    
18 
19    @Around("performanceTargets()")
20    public Object logPerformanceStats(ProceedingJoinPoint joinpoint) {
21        try {
22            long start = System.nanoTime();
23            Object result = joinpoint.proceed();
24            long end = System.nanoTime();
25            logger.info(String.format("%s took %d ns", joinpoint.getSignature(), (end - start)));
26            return result;
27        } catch (Throwable e) {
28            throw new RuntimeException(e);
29        }
30    }
31}

Here the pointcut expression –

view sourceprint?
1@Pointcut("execution(@org.bk.annotations.PerfLog * *.*(..))")

selects all methods annotated with @PerfLog annotation, and the aspect method logPerformanceStats logs the time taken by the method calls.

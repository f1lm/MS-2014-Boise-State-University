My $2375 Amazon EC2 Mistake

December 30, 2014/32 Comments/in DevBlog, DIY, Frontpage Article, ruby on rails /by Admin

Forward

Over the holidays, I opted to try to teach myself Ruby & its companion Rails.

Turns out, there are a great resources scattered about the web that can really help you drill down into rails and learn the nitty gritty details of how it all works.

I spent my time with Rails & Zombies, then I opted to read through the Rails tutorial. Unfortunately the Rails tutorial is pretty bland so about half way through I decided to snoop around to see what my options were. Somehow, almost by chance I ended up a subscriber on Lynda. Lynda only has one Rails tutorial, but its pretty in depth and is backed by a five hour Ruby tutorial.

Surprisingly I felt like although the Rails tutorial at Lynda was only 12 hours long, it had a ton of quality content.

But promoting Lynda isn’t what this post is about. This post is about what happened when I actually tried to apply my newly gained Rails knowledge.

The Idea

In order to apply my newly earned Rails knowledge, I opted to build a Yelp clone. It would be a simple website, up on Heroku running on Rails with a PostgreSQL back-end. Users would be able to sign-up, log in and post reviews. Signed in users could also add restaurants. Users that opted to remain anonymous  would still be able to view restaurants and reviews, but could not contribute any content to the site.

In order to accomplish this I needed a few basic things:

Static Pages:

  • About
  • Contact

Models (Database Tables):

  • Users
  • Restaurants
  • Reviews

Relationships (within tables)

  • Restaurants can have many reviews, but reviews can only have one restaurant
  • Users can have many reviews, but a review can only have one user

Schema:

  • Users have a username, email and hashed password. They also are related to reviews by a review_id.
  • Restaurants have a name, location, phone and website. They are related to reviews by a review_id.
  • Reviews have a title, integer rating (1-5 stars), and a text field. They are related to restaurants and users.

I was able to whip up the basic components of this site pretty quickly. A “rails generate new…” to get the project running, followed by a “rails controller Pages about contact”, and a few “rails scaffold…” commands.

Shortly afterwards, I edited the routes.rb in config to set the default page to restaurants, and I set up some pretty routes for direct URLS. I than realized my project was ugly, so I added bootstrap to my gem file and hacked away with some CSS (its still ugly, mind you).

Eventually, I opted to git push heroku master – so I could test online. The site, http://shriek.herokuapp.com looked fine and functioned fine. However, I noticed that the actual Yelp site also let you attach pictures to your reviews. I figured pictures couldn’t be too hard to add.

I made myself a picture model, and found a gem – carrierwave which handles uploads from a GUI. Thanks to carrierwave I was able to get my site online with some image functionality. But I couldn’t store the images on Heroku. So what where my other options?

Amazon was offering a 1 year free trial of AWS, with a limited cap on uploads. I figured I’d give it a shot, hook into S3 and host my images there. Actually hooking into the API was very straightforwards, and didn’t take more than an hour to set up. I knew my API key needed to be safe, so I installed the Figaro gem (a rails API key security gem, which typically works great), and trusted it to keep my API key off of git when I pushed. I opened the console and git push origin master to send the new version of my app, Shriek to Heroku.

Figaro pops up on the command line as usual, but this time instead of saying “created application.yml, created .gitignore” It just said “created application.yml”. Hmm, how weird I thought. I figured nothing was wrong, and it was just a message that didn’t popup. My code got pushed to GitHub and I went to check it out.

When I got to GitHub, I checked my application.yml, and sure enough it was online with my API keys… Crap!

I reverted the last few commits, and deleted all traces from GitHub. I was able to clean it up within about 5 minutes and no one else knew about the repo. After a close call, I went to bed.

Oops.

When I woke up the next morning, I had four emails from Amazon AWS and a missed phone call from Amazon AWS. Something about 140 servers running on my AWS account. What? How? I only had S3 keys on my GitHub and they where gone within 5 minutes!

Turns out through the S3 API you can actually spin up EC2 instances, and my key had been spotted by a bot that continually searches GitHub for API keys. Amazon AWS customer support informed me this happens a lot recently, hackers have created an algorithm that searches GitHub 24 hours per day for API keys. Once it finds one it spins up max instances of EC2 servers to farm itself bitcoins.

Boom! A $2375 bill in the morning. Just for trying to learn rails.

Lucky for me, I explained my situation to Amazon customer support – and they knew I wasn’t bitcoin mining all night. Amazon was kind enough to drop the charges this time!

I learned a valuable lesson here though. Don’t trust .gitignores and gems like Firago for keeping your data safe, these may work as a first layer of protection – but should be backed up by another level of security. Open source is awesome, but if you are dealing with anything that can be scaled up to thousands of dollars per hour – at least store it in a private repo if not on your local machine. And lastly, if your API keys get out on the web – revoke them and regenerate them. Its not worth the risk, even if it was only five minute online.

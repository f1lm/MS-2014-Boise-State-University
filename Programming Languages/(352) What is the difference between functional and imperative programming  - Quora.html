    Imperative programming involves writing your program as a series of instructions (statements) that can actively modify memory (variables, arrays). Imperative programming focuses on how, in the sense that you express the logic of your program based on how the computer would execute it.

    Functional programming involves writing your program in terms of functions and other mathematical structures. A function is just a rule that maps input elements to output—unlike an imperative procedure or subroutine, it can't rely on outside state or have any side-effects. Functional programming concentrates on the what, trying to let you specify the logic of your program as close to actual logic as possible rather than basing its semantics on how a computer would execute it.

    Fundamentally, functional programming is a different basis from imperative programming. It provides a completley new foundation for writing programs and requires a different way of thinking. However, there are some advantages: functional languages tend to be more expressive, so you end up writing less code; they tend to be safer so that more bugs are prevented by the language ahead of time and they have some features impossible in normal languages because side-effects are controlled.

    For this answer, I'll just focus on my last point, using Haskell as an example. Haskell is a so-called "pure" functional language. This does not mean it allows no side-effects; rather, it means it allows you to control side-effects. In a sense, side-effects are a first-class citizen: you can write code that guarantees its inputs do not have side effects, or you can write code that can have side-effects.

    This control of side-effects lets Haskell do some incredible things. A few examples include stream fusion, which automatically rewrites multiple loops into one; software transactional memory, which provide a database-like model for sharing memory using transactions and determinisitic parallelism which is exactly what it sounds like: you can evaluate a computation in parallel over n cores and be guaranteed that you still get the same answer. No race-conditions, no deadlock: nothing! It's actually pretty incredible.

    On top of this, the parallel strategies let you evaluate code in parallel without modifying the original code. No need to stick annotations everywhere or use different functions. You can ask the code to be evaluated in parallel from a different module. I wrote a bit more about this idea here: Could you please give me one motivating example that shows functional programming is better than imperative programming when writing concurrent programs?, which I think is worth reading.

    Functional Programming VS Imperative:
    1) Functional programming treats everything as a function. The key point is it does not have a state associated with it i.e it is immutable.
    So, the next best thing in terms of its usage is concurrent programming. As concurrent operations can be performed much easily without having to worry about state changes.
    Having to learn functional programming paradigms lets you understand the common semantics and syntax that most of the programming languages use.

    2) Imperative programming is procedural programming which is more focussed on how things are done. It has steps associated with it to perform a task, states and immutable objects.

    Object oriented programming is a very common imperative style of programming.Here everything revolves around objects which are real life entities. It has the excellent concepts of encapsulation and abstraction which are the building blocks of object oriented programming style. 

    Nowadays, people are focussing on hybrid languages which use the best of both approaches. One such example is SCALA which is a hybrid language.

    Functional programming relies primarily on data structures and functions.  Functions are applied against inputs to produce outputs.  Functions are themselves legitimate inputs and outputs.  It generally avoids side effects by only operating exclusively on the arguments passed into a function and nothing else.  Solving a problem deals with discovering what transformations are necessary to bring about revisions.  That is, data is not actually changed in as much as it is actually replaced or superceded by more current data.

    Imperative programming is less concerned with side effects and state changes.  Functions take arguments and produce outputs but, additionally, are allowed to effect changes in other parts of the system (e.g. produce side effects).  Solving a problem deals with applying changes to the existing data.  The data is not as much replaced as it is actually modified.

    Both paradigms deal with developing a solution as a series of steps applied against the data, they just take different routes to get there.

    Having worked with both paradigms I am more excited about functional programming.  It offers the function up as the most fundamental primitive instead of objects; in turn, I think once you understand how to use it, it offers a simpler way of thinking about problem solving.

    As I read through the other answers, I felt like the one unanswered question was, "How does one change state in FP?" This is a key thing to learn.

    In imperative programming you change state by changing a variable. You do an assignment, or use a loop construct, or do a branch to some code that does the same. If you're used to imperative programming (such as from Java, C++, etc.), this is a tough lesson to learn (at least it was for me): In FP you change state by calling a function. Now, it's not necessarily a different function from the one you're in. I'll explain. Also, you branch by using a condition. Conditions are how you selectively change state (using "if-then-else"-like logic. Not that different from structured programming in that regard).

    Within FP circles they create a subtle distinction between iterative and recursive program flow. In imperative programming circles, recursion just means a function that calls itself, with a "bootstrap condition" that ends the recursion. In FP, often the only way to change state is through what imperative programmers would call "recursion." However, in FP circles they may still talk about iterative program flow, even though they're talking about a function that calls itself. The difference they make is that iterative flow typically (though not always) has no state "waiting behind it" for it to return. Like this example:

    1
2
3
4

    (define (count-to-ten a)
       (cond ((= a 10) a)
             (else (display a)
                   (count-to-ten (+ a 1)))))


    The above code executes like (if we start with a=1):

    (count-to-ten 1)
    1 <-- this comes from the "display" call
    (count-to-ten 2)
    2
    (count-to-ten 3)
    .
    .
    .
    10 <-- returned by bootstrap condition, not from "display" call.

    In FP circles you may hear about "tail call optimization." Technically this means that the runtime environment doesn't put the inner call to "count-to-ten" on the stack as each call is made. This saves memory from being wasted by calls that have nothing "waiting behind them" to execute.

    What's called "recursion" is like this:

    1
2
3

    (define (dup a)
       (cond ((null? a) '())
             (else (cons (car a) (dup (cdr a))))))


    Here, we have a "cons" call waiting on "dup" to return. (This function creates an exact copy of a list it's given.)

    You'll notice in both the "iterative" and "recursive" examples a bootstrap condition is used to end the process. It's still essential. Otherwise, the function will go into an infinite loop. However, it can be seen as no different from "for" and "while" loop conditionals, except for the fact that a bootstrap condition becomes "true" when for/while conditionals become "false," when the looping process is ended.

    Another way to change state is with expressions like "define" or "let":

    1
2
3

    (define (count a)
       (define b (+ a 10)) ; In Scheme, 'b' cannot be redefined inside this scope.
       (display b))


    or

    1
2
3

    (define (count a)
       (let ((b (+ a 10)))
          (display b)))


    Interestingly, I learned that the above "let" expression can be (and perhaps is) implemented like this:

    ((lambda (b) (display b)) (+ a 10))


    So, it doesn't violate FP principles. If "let" is not implemented as a primitive, it's done as a macro.

    In Scheme, it does a compile check on "define" to make sure you're not redefining the same variable inside an enclosing scope. So it implements the same principle as what Victor was talking about: y = f(x), and will always be a placeholder for f(x), at least within that scope.

    I don't know if this is still true, but when I took CS over 20 years ago, I remember that it used to be explained that even binding a variable to a value could be seen in functional terms, that saying something like (define a 10) could be expressed as:

    a = f(x), where f(x) always equals 10

    Where that's inconsistent (or at least it's inconsistent inside Scheme and other Lisp's) is that you can't evaluate:

    (a)


    in a typical environment and get back "10." Instead it will tell you, "No function bound to a." Scheme binds symbols to atoms or lists, and in Scheme numbers are considered atoms, not lists with executable code (not functions). To get back the value of "a" you just evaluate the symbol:

    1
2

    > a
    10 <-- response from evaluator


    I think the other answers accurately describe the difference between imperative and functional programming, that imperative is about "steps," and functional is about "expressing truths." You'll notice in the examples I used there was a little imperative programming (with "display"). This is because in most (or all) FP implementations to date the display is not programmed functionally, so it has to be done imperatively. Same goes for loading and saving code, loading/saving data, etc.

    Yes, functional programming is about functions. But imperative programming languages also have functions.

    Here's the obvious difference: in functional programming you don't have variables of which you can change the value. That is because writing

    y = f(x)

    does not mean that the variable "y" gets assigned the value of "f(x)", it means that "y" is to all practical purposes is a synonym for "f(x)". In other words, before that statement "y" was undefined; starting with that statement, and for all eternity, "y" will have the value "f(x)".

    This behaviour makes functional languages behave much more like mathematics, and it becomes much easier to reason about a piece of code. In ordinary programming languages it is hard to decide whether a variable has the same value at two different points, and therefore whether you can exchange two statement or lift a statement out of a loop (for efficiency). In functional languages this is much easier.

    Of course there are problems with functional languages. Since you can not change an object (that's called side-effects) you also have to make a copy. Ok, now image you have an array that takes up 1/3 of your 300 Gb computer memory, and you change one byte...... Now you have to make a complete copy of it. And if the array took up 2/3 of your memory you're stuck.

    Functional language people also have never heard of such hardware concepts as registers or caches. Sometimes you want to overwrite a variable because it's in a register and therefore extremely efficient. A functional language simply can't express this, so it puts a heavy burden on the compiler to figure out when it's safe to overwrite data.

    I don't think functional compilers are remotely that clever.

    Functional programming is about writing a set of, well, functions that carry out a particular calculation.

    So, you might define a function called : 'Square' along the lines of:

    > Square (X) := X * X

    Then you might run the function by saying:

    > Square (9)

    ... and get the answer

    > 81

    Imperative programming is about running operations in sequence. If we go to a simple imperative language, say, 1980s BASIC. You can't write the function. You have to write out the code for the square function each time you use it.

    10 LET A = 9
    20 LET B = A * A
    30 PRINT B


    Obviously, SQUARE is a simple function... more complex functions in functional languages are typically defined by recursion. So, let's consider how we define the function 'MIN' in a simple functional language like Orwell.

    You need to know a bit of Orwell notation - x is a single number... xs is a list of numbers... and x:xs is the list that starts with x, and then has the members of xs... so x = 1, xs = [2,3,4,5] means that x:xs = [1,2,3,4,5]

    You define a function called MIN as follows:

    MIN (x) = x
    MIN (x:xs) = x, if x < MIN(xs)
                         MIN (xs) otherwise

    Calling MIN [1,2,3,4,5] requires a lot of recursive calling, as the interpreter realises it has to evaluate MIN [2,3,4,5] and to do that MIN [3,4,5], and to do that MIN [4,5] and to do that MIN [5].... before it can pass the results back up the chain of calls and get out the answer '1'



    In an imperative language, you write something like:

    x = [1,2,3,4,5]
    m = x.1 (the first element in the array)
    for n = each member of x
    if m > n, m := n
    end for

    Obviously, the syntax varies.



    Now, most modern languages are hybrids.... if you regularly needed to call a min function (and for some reason, it wasn't in the language as a native call), you'd write a function called Min, rather than copying those lines of code out...


    The key difference is that, in a hybrid or an imperative language you can do stuff like setting global variables in the middle of the function...

    ... in a purely functional language there are no global variables. You have to pass everything into the function arguments, so all MIN(X) can do is return the answer... in strong functional languages, it can't even modify X. (Or put another way, all calls are by value, rather than reference.)

    In addition, in a pure functional language there are no loop constructs - you can't do 'WHILE'... or 'FOR'... or anything like that.

    You can typically do 'IF' and 'CASE'.


    To be honest, functional languages are far more useful if they have imperative features... and vice versa.

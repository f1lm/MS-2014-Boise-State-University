Data Types

Scala's data types are based on Java's, but all types are Objects

Scala uses many of the exact same data types as Java, and both languages use the same amount of memory and precision for their common types. However, each type in Scala is an Object, unlike Java, where there are primitive types. Therefore, you can call methods on them directly without the need of a wrapper class. The types in Scala are detailed below:

Boolean True or false
Char A unicode character held in 16 bits
String A series of Chars
Null An empty reference
Byte A signed value held in 8 bits
Short A signed value held in 16 bits
Int A signed value held in 32 bits
Long A signed value held in 64 bits
Float A single-precision float held in 32 bits
Double A double-precision float held in 64 bits

In addition to these familiar types, Scala has a few more types worth noting:

Nothing Subtype of every other type
Any Supertype of every other type
AnyRef Supertype of all reference types

The need for these types arises from the limitations of the Java Virtual Machine on which Scala runs. As mentioned before, ALL types in Scala are an object, unlike in Java.

To accomplish this, all Objects in Scala descend from Any, including Java’s “primitives” and Java’s “Objects.” AnyVal extends Any and is used as a superclass to everything that was considered a primitive in Java. AnyRef holds the remainder, and is essentially equivalent to the Object class in Java. Interestingly, Any and AnyVal were recently unsealed in Scala; programmers can now extend them if desired.

The use of these types should be familiar to any Java user. For example, integer arithmetic works exactly the same. Division that yields a fractional number will be truncated. The biggest difference is that the use of autoboxing is no longer necessary. For example, in Java, to convert a char to an integer, you’d have to make use of the wrapper class Character like this:

              
				int myInt = Character.getNumericValue(myChar);

But in Scala, you can call a method on myChar, resulting in a simple conversion:

              
				myChar.asDigit

First Class Functions

Scala allows the passing of functions as a value

In another large deviation from Java, Scala has allowed the passing of functions in the same way as other types. This is somewhat similar to using function pointers in C. More formally, a first-class function in Scala can be used, constructed, and typed just like “normal” types. Take a look at the following example:

            
				def multByTwo(i : Int) = i * 2
				val secondaryFunction : Int => Int = multByTwo
				println(secondaryFunction(2))

This will result in an output of 4. Note the function multByTwo being used as a value in secondaryFunction. This allows a great deal a freedom; functions can now be put in containers, functions can actually be returned by other functions, and so on!

The example above only shows the function literal being passed as a parameter, but it is also possible to reference variables defined elsewhere. For example:

            
				(myInt: Int) => myInt * outsideVar

Now, this function refers to a variable outsideVar that must have been declared elsewhere, while myInt is bound. If this function is used without outsideVar existing, a run-time error will occur. However, if outsideVar exists, it will evaluate as expected. The function value (which is an object in Scala) is called a closure. This name comes from the function literal being closed when the bindings of free variables such as outsideVar are determined.

Scoping

Scala continues the modern trend of static scoping

Scala is statically scoped, like most recent languages. The scope of any variable is determined by the blocks, or nesting braces in the source. This differs from dynamic scoping, which instead makes variables available based on the nesting of function calls at run-time and cares nothing for their location.

Dynamic scoping breaks the idea of encapsulation, and would be inappropriate in an Object-Oriented language such as Scala. If passing a variable to hold state information becomes impractical due to the complexity of the program, the traditional approach is to make the variable global. This should be familiar to most Object-Oriented programmers.
